WHAT WAS BUILT
================================================================================

A comprehensive full-frame integration test framework for the Taito X graphics
pipeline, specifically testing the sprite→colmix path:

  1. X1-001A Phase 2 (sprite scanning + rendering)
  2. taito_x_colmix (palette lookup + priority compositing)

Framework consists of:
  - Python reference model (gate4_model.py)
  - Test vector generator (generate_gate4.py)
  - Verilator C++ testbench (tb_gate4.cpp)
  - Build automation (Makefile)
  - Complete documentation (README.md, QUICKSTART.md, etc.)

================================================================================
ARCHITECTURE
================================================================================

Gate4 pipeline simulation:

  Sprite Y RAM (0x180 words)
  Sprite Code/Attr RAM (0x2000 words)
  GFX ROM (0x40000 words)
    ↓
  X1-001A Phase 2 (sprite scanner + renderer)
    ↓
  Per-pixel (valid, color_idx)
    ↓
  Palette RAM (0x800 words, xRGB_555)
    ↓
  taito_x_colmix (palette lookup)
    ↓
  Final RGB (5 bits per channel)

Key classes:

  Python:
    - Gate4Model         — Full pipeline model (wraps X1001APhase2)
    - X1001APhase2      — Sprite rendering (from x1001a_model.py)

  C++:
    - DUT               — Verilator wrapper for X1-001A RTL
    - main()            — Test loop: load vectors, execute ops, validate

================================================================================
USAGE
================================================================================

Quick start:
  1. cd chips/taito_x/vectors/gate4_integration/
  2. make              (full pipeline: generate → build → run)

Step-by-step:
  1. python3 generate_gate4.py     (generate test vectors)
  2. make build                     (compile with Verilator)
  3. make run                       (execute testbench)

Manual compilation:
  verilator --cc --exe tb_gate4.cpp --top-module x1_001a \
    ../../rtl/x1_001a.sv -Mdir obj_dir
  cd obj_dir && make -f Vx1_001a.mk
  ./Vx1_001a gate4_integration_vectors.jsonl

TEST COVERAGE
================================================================================

Test 1: Simple Sprite
  Input: Tile 1 (solid color 7), color attr 5, at screen (100, 112)
  Expected: 16×16 cyan pixels (palette[87] = xRGB_555)
  Validates: Basic sprite rendering, palette lookup

Test 2: Multi-Sprite Priority
  Input: Sprite 0 (tile 1, color 3, red) at (50, 100)
         Sprite 1 (tile 2, color 4, blue) at (60, 105)
  Overlap: Expected red (sprite 0 higher priority)
  Validates: Sprite priority, overlapping compositing

Test 3: Clipped Sprite
  Input: Tile 3 (gradient), at right edge
  Expected: Only visible portion renders
  Validates: Screen edge clipping, boundary conditions

Test 4: Palette Check
  Input: Tile 4 (striped), color 2 (green)
  Expected: Each column has distinct palette color
  Validates: Palette lookup correctness, color attributes

Test 5: Transparent Pixels
  Input: Tile 5 (checkerboard: pen=0 and pen=15), color 6
  Expected: Opaque pixels (pen=15) show magenta
            Transparent pixels (pen=0) show background
  Validates: Transparency (pen=0), background visibility

KNOWN LIMITATIONS & FUTURE WORK
================================================================================

Current limitations:

  1. Palette BRAM not yet instantiated in testbench
     - palette_write operations logged but not driven to RTL
     - Would require adding BRAM primitive to testbench

  2. No RGB output capture yet
     - check_pixel validated against Python model, not RTL
     - Full integration requires instantiating taito_x_colmix

  3. Background tilemap stubbed
     - Currently defaults to palette[0] for non-sprite pixels
     - Could be extended with background rendering

  4. Single COLOR_BASE value
     - Tests use COLOR_BASE=0
     - Could test multiple game-specific offsets

Future improvements:

  Immediate (to complete gate4):
    - Add palette BRAM to testbench

    - Capture rgb_r/rgb_g/rgb_b outputs
