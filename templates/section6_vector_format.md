# Section 6 — Test Vector JSON Schema

## Overview

Test vectors for gate1 simulation are stored as JSON Lines (`.jsonl`) files —
one JSON object per line, each representing one clock cycle's worth of inputs
and expected outputs.

Vectors live in:
```
chips/<CHIPNAME>/vectors/<testname>.jsonl
```

## JSON Lines Schema

Each line is a single JSON object with the following fields:

```json
{
  "t":       <integer>,          // REQUIRED: simulation time in clock cycles (monotonically increasing)
  "comment": <string>,           // OPTIONAL: human-readable annotation for this cycle
  "inputs":  { <signal>: <value>, ... },   // REQUIRED: all input signals this cycle
  "outputs": { <signal>: <value>, ... },   // REQUIRED: all expected output signals this cycle
  "mask":    { <signal>: <value>, ... },   // OPTIONAL: bitmask for outputs to check (default: all bits)
  "flags":   { <flag>: <bool>, ... }       // OPTIONAL: special simulation control flags
}
```

### Field Definitions

#### `t` (required, integer)
Simulation time in clock cycles from reset release (t=0 is the first cycle after
rst_n deasserts). Must be monotonically increasing. Gaps are allowed — the
simulator advances the clock without checking outputs for skipped cycles.

#### `comment` (optional, string)
Human-readable description of what this vector tests. Used in failure messages.
Example: `"first pixel after hblank"`

#### `inputs` (required, object)
Map of signal name to value for all DUT inputs this cycle.
- Boolean signals: `0` or `1`
- Multi-bit signals: decimal integer, or `"0xHH"` hex string, or `"0bBBBB"` binary string
- Signals not listed retain their previous cycle's value (sticky)
- Signal names must exactly match the DUT port names

```json
"inputs": { "clk": 1, "rst_n": 1, "en": 1, "sel": "0b01" }
```

#### `outputs` (required, object)
Map of signal name to expected value this cycle. Simulator compares DUT output
against this value after the clock edge. Any mismatch = gate1 FAIL.
- Same value encoding as `inputs`
- Signal names must exactly match the DUT port names

```json
"outputs": { "count": 5, "valid": 1 }
```

#### `mask` (optional, object)
Per-signal bitmask. Only bits where mask bit = 1 are compared. Useful for
"don't care" fields (e.g., checking only the lower nibble of a byte output).

```json
"mask": { "pixel_data": "0x0F" }
```

If `mask` is omitted for a signal, all bits are compared (mask = all ones).

#### `flags` (optional, object)
Special simulation control. Supported flags:

| Flag | Type | Meaning |
|------|------|---------|
| `"reset"`    | bool | Assert reset this cycle (overrides rst in inputs) |
| `"skip"`     | bool | Skip output comparison this cycle (inputs still applied) |
| `"stop"`     | bool | Stop simulation after this cycle (pass if all prior checks passed) |
| `"edge"`     | string | `"posedge"` or `"negedge"` — which clock edge to sample on |

## Example Vector File

```jsonl
{"t": 0, "comment": "reset release", "inputs": {"rst_n": 0, "en": 0}, "outputs": {"count": 0}, "flags": {"reset": true}}
{"t": 1, "comment": "first cycle after reset", "inputs": {"rst_n": 1, "en": 0}, "outputs": {"count": 0}}
{"t": 2, "comment": "enable counter", "inputs": {"en": 1}, "outputs": {"count": 0}}
{"t": 3, "comment": "count increments", "inputs": {}, "outputs": {"count": 1}}
{"t": 4, "inputs": {}, "outputs": {"count": 2}}
{"t": 5, "inputs": {}, "outputs": {"count": 3}}
{"t": 10, "comment": "disable counter", "inputs": {"en": 0}, "outputs": {"count": 8}}
{"t": 11, "comment": "count holds when disabled", "inputs": {}, "outputs": {"count": 8}}
{"t": 12, "comment": "mid-run reset", "inputs": {"rst_n": 0}, "outputs": {"count": 0}}
{"t": 13, "inputs": {"rst_n": 1, "en": 1}, "outputs": {"count": 0}}
{"t": 14, "inputs": {}, "outputs": {"count": 1}, "flags": {"stop": true}}
```

## 3-Tier Testing Strategy

See `vectors/README.md` for the full strategy. In brief:

| Tier | Vectors | Coverage |
|------|---------|----------|
| Tier 1 | Reset and power-on behavior | Gate1 minimum viable |
| Tier 2 | Functional state transitions | All documented chip states |
| Tier 3 | MAME-derived corner cases | Cycle-accurate edge cases from MAME src |

## MAME Ground Truth

Tier 3 vectors are generated by running the MAME emulator with a reference ROM
and logging all chip pin states cycle-by-cycle. The `chips/<CHIPNAME>/vectors/`
directory should contain:
- `tier1_reset.jsonl` — basic reset/enable
- `tier2_functional.jsonl` — state machine coverage
- `tier3_mame.jsonl` — MAME-derived cycle-accurate vectors

Gate4 runs tier3 vectors and compares against MAME output. Expected first-pass
pass rate: 65–75% (25–35% failure rate is normal — iterate on the prompt).
