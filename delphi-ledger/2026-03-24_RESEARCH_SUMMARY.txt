KEY FINDINGS:
================================================================================

1. SOURCE CODE AVAILABILITY: ❌ NOT PUBLIC
   └─ Coin-Op Collection FPGA source is closed-source
   └─ Only compiled .core files (Analogue Pocket) available on GitHub
   └─ Pramod's MiSTer Y-Unit source never released to public

2. SCANLINE BUFFER APPROACH: ⚠️ VIABLE ONLY FOR 2 GAMES
   └─ Works: Smash TV, Total Carnage (direct framebuffer, no staging)
   └─ Fails: All other Y-Unit games (use hidden VRAM rows for sprite prep)
   └─ Root cause: Games assume unified address space; scanline buffer
      splits BRAM (visible rows 0-1) from SDRAM (hidden rows 256+)
   └─ Result: CPU reads stale sprite data, animation corrupts

3. STAGING RAM ARCHITECTURE: 🔴 CRITICAL BLOCKER
   └─ 6 of 8 Y-Unit games use off-screen "staging" VRAM rows
   └─ Sprite composition happens in rows 256-511 (hidden from display)
   └─ Game assumes simultaneous:
      • CPU reads from staging (row 300+) for animation frame lookup
      • DMA writes to visible framebuffer (rows 0-255) for display
   └─ Scanline buffer breaks this by splitting memory into two domains

4. BANDWIDTH REALITY: 📊
   └─ Pramod spent 6+ months optimizing MiSTer implementation
   └─ Tried: 125 MHz SDRAM, CAS=2, careful arbitration
   └─ Peak demand: 65-95 MB/s (DMA + CPU + Autoerase + line output)
   └─ MiSTer available: ~200 MB/s @ 125 MHz 16-bit SDRAM
   └─ Problem: Contention during HBlank locks entire VRAM
   └─ Solution (Coin-Op): 128 MHz SDRAM + 32-bit burst DMA writes

5. FEASIBILITY BY PLATFORM:
   ✅ Analogue Pocket (52K ALM, 128 MHz SDRAM): ALL 8 GAMES (RC6 released)
   ⚠️  MiSTer DE10-Nano (40K ALM, 125 MHz SDRAM): 2 GAMES ONLY (Smash TV + Total Carnage)
   ❌ Full Y-Unit on MiSTer: NOT FEASIBLE without larger FPGA or external DDR

TECHNICAL ARCHITECTURE:
================================================================================

TMS34010 CPU CORE:
• 32-bit internal, 16-bit external (CPU-to-RAM)
• 1-bit addressing (converts to byte addresses for DRAM)
• Integrated memory controller + shift buffer
• Integrated GPU (FILL instruction for bulk clear, MOVE with transparency)
• Implementation: 7000+ lines Verilog (per Pramod's blog)

Y-UNIT MEMORY:
• VRAM: 512 KB unified address space (256 KB Palette + 256 KB Bitmap)
• DMA: 32-bit fetch from Image ROM, 6-bit pixel live deswizzling
• DMA Modes: 9 options (palette swap, transparency, solid color fills)
• Autoerase: Hardware PLD (most games) OR GPU FILL (Strike Force, T2)
• Pixel clock: 8 MHz (41 ns/pixel throughput requirement)
• Resolution: ~400×256 @ 6bpp

VRAM ARBITRATION (4-way):
1. DMA (highest priority during game loop): 32 MB/s
2. CPU (random reads/writes): 8-16 MB/s
3. Autoerase PLD (during HBlank): 15-25 MB/s
4. Line buffer output (CRT): 8-16 MB/s
